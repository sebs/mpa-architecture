# ADR-001: Multi-Page Architecture Over Single-Page Application

**Issue**

We must determine the fundamental navigation model for the application before any development begins. This decision affects every subsequent architectural choice including state management, code organization, build tooling, and deployment strategy. The industry default has shifted toward Single-Page Applications, but this default carries assumptions about application complexity and user experience goals that may not align with our principles.

**Decision**

Adopt Multi-Page Architecture. Each view is a separate HTML document served independently. Navigation occurs through standard anchor links with full page loads. The browser handles all routing natively.

**Status**

Decided

**Group**

Presentation

**Assumptions**

- Modern browsers handle page loads efficiently with HTTP/2 and aggressive caching
- Users understand and expect web navigation patterns (back button, bookmarks, link sharing)
- Search engine optimization matters for discoverability
- Development team is comfortable working without framework abstractions
- The application does not require sub-second view transitions that only client-side routing can provide

**Constraints**

- JavaScript execution context is lost on each navigation
- Shared state must persist through storage mechanisms rather than memory
- Common UI elements (header, navigation, footer) must be duplicated across HTML files or generated server-side

**Positions**

1. **Single-Page Application with framework** — Use React, Vue, or similar to manage all rendering and routing client-side. Industry standard approach with extensive tooling and community support. Rejected due to framework complexity and lock-in.

2. **Single-Page Application without framework** — Implement custom client-side routing and view management in vanilla JavaScript. Avoids framework dependency but recreates framework problems. Rejected due to recreating framework complexity without its tooling.

3. **Multi-Page Application with partial hydration** — Serve static HTML pages but enhance with JavaScript "islands" for interactivity. Compromise approach gaining popularity. Rejected due to build tooling requirements.

4. **Multi-Page Application pure** — Each page is complete and independent. Browser handles navigation. JavaScript enhances but does not control navigation. This is our selected position.

**Argument**

Single-Page Applications solve problems we do not have while creating problems we wish to avoid. SPAs exist primarily to enable rapid view transitions and complex in-memory state management. Our application prioritizes different values: simplicity, debuggability, and alignment with web platform behavior.

The costs of SPA architecture are substantial: routing libraries, state management, code splitting, server-side rendering for SEO, hydration complexity, and framework lock-in. These costs compound over time through maintenance burden and upgrade cycles.

MPA architecture provides immediate benefits without ongoing costs. Browser navigation works correctly by default. Each page can be developed, tested, and cached independently. Search engines index content without special tooling. The back button works. Bookmarks work. Link sharing works. These behaviors require zero code.

The primary SPA advantage—fast view transitions—matters less than commonly assumed. Users have decades of experience with page loads. A 200ms page load is not a poor experience; it is the expected web experience. Optimizing for 50ms transitions while accepting framework complexity is a poor trade.

**Implications**

- State that must survive navigation requires explicit persistence (ADR-003, ADR-009)
- Cross-tab coordination requires explicit messaging (ADR-007)
- Common UI elements will be duplicated; consider templating during development if repetition becomes burdensome
- Service Worker caching becomes essential for performance (ADR-008)
- Each page connects to the same IndexedDB database; no per-page initialization overhead

**Related Decisions**

- ADR-002: No JavaScript Frameworks (reinforces MPA choice)
- ADR-003: IndexedDB as Primary Data Store (provides persistence across navigations)
- ADR-007: Broadcast Channel (solves cross-tab coordination)
- ADR-008: Service Worker Caching (makes repeated page loads fast and offline-capable)
- ADR-009: URL Parameters as State (defines where navigation state lives)

**Related Requirements**

- Application must be indexable by search engines
- Application must support standard browser navigation (back, forward, bookmarks)
- URLs must be shareable and produce consistent views
- Application must function without JavaScript for basic content viewing

**Related Artifacts**

- Project structure documentation
- Page template specifications
- Navigation flow diagrams

**Related Principles**

- Work with the browser, not against it
- Progressive enhancement over application shells
- Simplicity over capability

**Notes**

Team discussion considered whether "SPA-like feel" was a goal. Consensus reached that this framing was misguided. We are building a website with database capabilities, not an application that happens to run in a browser. Page loads are acceptable and expected.
