# ADR-002: No JavaScript Frameworks

**Issue**

JavaScript frameworks provide component models, reactivity systems, and ecosystem tooling that accelerate certain types of development. They also introduce runtime overhead, build complexity, and framework-specific patterns. Given our MPA architecture decision, we must determine whether frameworks provide sufficient value for page-level interactivity to justify their costs.

**Decision**

Use vanilla JavaScript with ES modules exclusively. No frameworks, component libraries, or virtual DOM implementations.

**Status**

Decided

**Group**

Technology

**Assumptions**

- ES modules are well-supported in target browsers
- Development team has sufficient JavaScript and DOM API knowledge
- Page-level interactivity requirements are moderate (data display, forms, filtering)
- We are not building complex interactive widgets (rich text editors, drag-and-drop interfaces)
- Long-term maintenance matters more than initial development speed

**Constraints**

- No JSX or framework-specific templating syntax
- No automatic reactivity; DOM updates require explicit code
- No pre-built component ecosystem; components must be authored or sourced individually
- Team members accustomed to frameworks may experience initial friction

**Positions**

1. **Full framework adoption** — Use React, Vue, or Svelte for all JavaScript. Provides component model, reactivity, and ecosystem. Requires build step and runtime. Rejected due to build step and runtime requirements.

2. **Lightweight framework** — Use Preact, Alpine.js, or similar minimal framework. Smaller runtime cost but still introduces abstraction layer and patterns. Rejected due to the added abstraction layer.

3. **Web Components** — Use custom elements and shadow DOM for componentization. Native browser feature but complex API and styling challenges. Rejected due to API complexity and styling challenges.

4. **Vanilla JavaScript with ES modules** — Direct DOM manipulation, native events, standard JavaScript. No abstraction layer. This is our selected position.

**Argument**

Frameworks solve the problem of managing complex, frequently-changing UI state across many interdependent components. Our MPA architecture eliminates most of this complexity by design. Each page loads fresh, queries data, renders once, and responds to user events. This is a fundamentally simpler model than the continuous reconciliation SPAs require.

The framework value proposition assumes:
- Deep component hierarchies with shared state
- Frequent re-renders based on state changes
- Complex event propagation across components
- Team members with varying JavaScript proficiency

None of these assumptions hold for our architecture. Our pages have shallow DOM structures with straightforward data binding. State changes (like adding a cart item) trigger explicit database writes and broadcast messages, not cascading re-renders. Events are handled at the page level, not bubbled through component trees.

Vanilla JavaScript has underrated advantages:
- Browser DevTools show actual running code
- No source maps needed; line numbers match
- No framework version upgrades or breaking changes
- No bundle size from runtime
- Skills transfer across projects regardless of framework fashion

The cost is more explicit DOM manipulation code. This cost is acceptable given our page complexity and is offset by debuggability gains.

**Implications**

- DOM manipulation patterns must be established and documented
- Common utilities may emerge; resist premature abstraction
- Templating happens through template literals or HTML string building
- Event delegation patterns useful for dynamic content
- Testing uses jsdom rather than framework-specific testing utilities

**Related Decisions**

- ADR-001: Multi-Page Architecture (reduces need for framework capabilities)
- ADR-004: Semantic HTML (structure provides styling hooks without component abstraction)
- ADR-005: No Build Step (vanilla JS requires no transpilation)
- ADR-010: Testing Approach (jsdom sufficient without framework test utilities)

**Related Requirements**

- Application must function in browsers without transpiled JavaScript
- Development must not require build step for iteration
- Code must be debuggable without source maps

**Related Artifacts**

- JavaScript coding standards document
- DOM manipulation patterns guide
- Module organization guidelines

**Related Principles**

- Simplicity over capability
- Debuggability over convenience
- Web platform over abstractions

**Notes**

Discussed whether TypeScript (without other frameworks) would be acceptable. Decided that TypeScript requires a build step for browser execution, violating ADR-005. Type checking can run separately as a development tool without producing build output, but runtime code must be valid JavaScript.
