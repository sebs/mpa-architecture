# ADR-003: DuckDB-WASM as Primary Data Store

**Issue**

Browser-based applications require data storage for structured information. Native browser storage options (localStorage, IndexedDB) have significant limitations for relational data and complex queries. We need a data layer that supports sophisticated queries without server connectivity, enabling true offline capability and eliminating network latency for data operations.

**Decision**

Use DuckDB compiled to WebAssembly as the primary data store. All structured application data resides in DuckDB tables, queried via standard SQL.

**Status**

Decided

**Group**

Data

**Assumptions**

- Target browsers support WebAssembly
- Data volume fits in browser memory (tens of megabytes typical, hundreds possible)
- SQL is an acceptable query interface for the development team
- Application requires relational data with joins and aggregations
- DuckDB-WASM project remains actively maintained

**Constraints**

- WASM binary adds several megabytes to initial page load
- Database must initialize on each page load (memory is not shared across navigations)
- Persistence requires explicit serialization to IndexedDB
- Browser memory limits constrain maximum data size
- No server-side query validation; all queries run client-side

**Positions**

1. **localStorage** — Simple key-value storage. Limited to 5MB. String-only values. No query capability. Synchronous API blocks main thread. Rejected due to capacity and capability limitations.

2. **IndexedDB directly** — Larger storage capacity. Supports binary data. Cursor-based API awkward for relational queries. No SQL. No joins without manual implementation. Rejected due to query limitations.

3. **IndexedDB with wrapper library** — Libraries like Dexie.js improve IndexedDB ergonomics. Still no true SQL or efficient joins. Rejected due to relational query limitations.

4. **SQL.js (SQLite in WASM)** — SQLite compiled to WebAssembly. Full SQL support. Mature and well-tested. However, optimized for transactional workloads rather than analytical queries.

5. **DuckDB-WASM** — DuckDB compiled to WebAssembly. Full SQL support. Optimized for analytical queries with columnar storage. Efficient aggregations and joins. Modern active development. This is our selected position.

**Argument**

The choice between SQL.js and DuckDB-WASM is the substantive decision; other options lack necessary capabilities.

DuckDB offers advantages for our use case:
- Columnar storage optimizes analytical queries (filtering, aggregation, reporting)
- Vectorized execution engine processes data efficiently
- Modern SQL dialect with useful extensions
- Active development with regular improvements
- Designed for embedded analytical use cases

SQL.js (SQLite) advantages:
- Longer track record in browser environments
- Larger community and more documentation
- Row-based storage better for single-record operations
- Smaller WASM binary

Our application leans analytical: displaying product listings with filters, calculating cart totals, generating summaries. These workloads favor DuckDB's design. The larger binary size is acceptable given Service Worker caching.

Both options require the same persistence strategy (serialization to IndexedDB) and have similar initialization patterns. Switching between them later would be feasible if needed.

**Implications**

- Team must be comfortable writing SQL
- Database schema design becomes important architectural artifact
- Persistence strategy needed for cross-session survival (ADR-006)
- Each page load initializes a fresh database instance from persisted data
- Query performance depends on appropriate indexing
- Memory usage monitoring may be needed for large datasets

**Related Decisions**

- ADR-006: IndexedDB Persistence (where DuckDB data survives)
- ADR-008: Service Worker Caching (caches WASM binary)
- ADR-001: Multi-Page Architecture (each page reinitializes database)

**Related Requirements**

- Application must function offline with full data access
- Data operations must not require network connectivity
- Complex filtering, sorting, and aggregation must be supported
- Application must handle datasets of at least 10,000 records performantly

**Related Artifacts**

- Database schema documentation
- Query patterns guide
- Data migration procedures

**Related Principles**

- Offline-first over network-dependent
- SQL as universal data interface
- Browser as complete runtime environment

**Notes**

Benchmarking recommended before committing to large-scale deployment. Test DuckDB initialization time and query performance with representative data volumes. If initialization exceeds 500ms consistently, may need optimization strategies (lazy loading, Web Worker execution).
