# ADR-003: IndexedDB as Primary Data Store

**Issue**

Browser-based applications require data storage for structured information. Native browser storage options have different characteristics and capabilities. We need a data layer that supports offline capability, structured data with relationships, and reasonable query performance without server connectivity.

**Decision**

Use IndexedDB as the primary data store. All structured application data resides in IndexedDB object stores. Use the native API directly first; a lightweight wrapper is acceptable if API friction warrants it.

**Status**

Decided

**Group**

Data

**Assumptions**

- Target browsers support IndexedDB (universal modern browser support)
- Data volume fits in browser storage quotas (typically generous, hundreds of MB to GB)
- JavaScript-based filtering and indexing is sufficient for query needs
- Application requires structured data with relationships
- Query complexity is moderate (no need for SQL joins or complex aggregations)

**Constraints**

- Query API is cursor-based and verbose; wrapper recommended
- No native SQL; complex queries require JavaScript implementation
- Transactions are required for multi-store operations
- Browser storage can be cleared by user or under storage pressure
- No server-side query validation; all operations run client-side

**Positions**

1. **localStorage** — Simple key-value storage. Limited to 5MB. String-only values. No query capability. Synchronous API blocks main thread. Rejected due to capacity and capability limitations.

2. **IndexedDB directly** — Larger storage capacity. Supports binary data. Object stores with indexes. Asynchronous API. Native browser feature requiring no additional code. This is our selected position.

3. **IndexedDB with wrapper library** — Libraries like Dexie.js improve IndexedDB ergonomics. Adds dependency but simplifies code. Acceptable alternative if raw API proves burdensome.

4. **SQL.js or DuckDB-WASM** — SQLite or DuckDB compiled to WebAssembly. Full SQL support. Adds several megabytes to page load. Requires serialization for persistence. Rejected due to complexity and payload size for moderate query needs.

5. **Remote server** — Store data on server, browser is cache. Adds server dependency. Rejected as contradicting offline-first architecture.

**Argument**

IndexedDB is the web platform's native solution for structured client-side storage. It offers significant advantages for our use case:

**Native to the browser.** No additional code to download, parse, or initialize. Available immediately on page load. No WASM binary or library overhead.

**Generous storage quotas.** Modern browsers allocate substantial storage (often based on available disk space). Far exceeds localStorage's 5MB limit.

**Structured data with indexes.** Object stores hold JavaScript objects directly. Indexes enable efficient queries on specific properties. Compound indexes support multi-property queries.

**Transactional guarantees.** ACID properties ensure data consistency. Operations complete fully or not at all.

**Asynchronous API.** Does not block the main thread. Integrates naturally with async/await patterns.

The primary criticism of IndexedDB is its verbose, callback-based API. This is mitigated through:
- Modern async/await wrappers (minimal code, no library needed)
- Dexie.js if API friction becomes significant (small, focused library)

For applications with moderate query complexity—filtering, sorting, paginating through hundreds or thousands of records—IndexedDB with JavaScript processing is sufficient. The overhead of loading a SQL engine in WASM is not justified.

**Implications**

- Team must understand IndexedDB concepts: databases, object stores, indexes, transactions
- Consider a lightweight wrapper for ergonomic improvements
- Schema design (object stores and indexes) is important architectural work
- Complex queries may require JavaScript post-processing
- Each page connects to the same database; no per-page initialization overhead

**Related Requirements**

- Application must function offline with full data access
- Data operations must not require network connectivity
- Filtering, sorting, and pagination must be supported
- Application must handle datasets of at least 10,000 records performantly

**Related Artifacts**

- Database schema documentation

**Related Principles**

- Offline-first over network-dependent
- Browser platform over external dependencies
- Simplicity over capability

**Notes**

If query complexity grows beyond moderate levels (frequent multi-table joins, complex aggregations, full-text search), reconsider introducing a query library or WASM-based solution. Start simple; add complexity only when justified by actual needs.
