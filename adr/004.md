# ADR-004: Semantic HTML with Classless CSS

**Issue**

HTML structure and CSS styling approach affect accessibility, maintainability, and development patterns. Modern practices often emphasize utility classes, CSS-in-JS, or preprocessors. These approaches can obscure document structure and require build tooling. We must establish conventions before authoring any markup.

**Decision**

Use semantic HTML elements as primary styling hooks. CSS targets elements and structural relationships rather than class names. Classes used only when element context cannot distinguish variants.

**Status**

Decided

**Group**

Presentation

**Assumptions**

- HTML5 semantic elements have sufficient variety for application needs
- Target browsers fully support HTML5 elements
- Accessibility is a primary concern
- Development team understands semantic element purposes
- Visual design can be expressed through element styling rather than arbitrary class application

**Constraints**

- Some visual variations may require restructuring HTML to distinguish contexts
- Cannot apply arbitrary one-off styles without adding markup or using inline styles
- Design system must align with available semantic elements
- Third-party embedded content may not follow conventions

**Positions**

1. **Utility-first CSS (Tailwind)** — Apply single-purpose classes directly in HTML. Rapid prototyping but HTML becomes verbose and presentational. Requires build step for optimization. Rejected due to build requirement and semantic obscurity.

2. **BEM or similar methodology** — Structured class naming conventions. Clear relationship between classes. Still relies heavily on class names rather than elements. Rejected due to class proliferation.

3. **CSS-in-JS** — Styles defined in JavaScript, scoped to components. Requires framework and build tooling. Rejected due to framework and build requirements.

4. **CSS preprocessors (Sass, Less)** — Extended CSS syntax with variables, nesting, mixins. Requires build step. Rejected due to build requirement.

5. **Classless CSS with semantic HTML** — Style elements directly. Use HTML structure as styling hook. Minimal classes for true variants. Plain CSS files. This is our selected position.

**Argument**

Semantic HTML provides a rich vocabulary of elements with distinct purposes: article, section, nav, aside, header, footer, main, figure, details, dialog, and many more. Each element has meaning to browsers, screen readers, and search engines. Styling these elements directly produces CSS that reads as "this is how articles look" rather than "this is what the card class does."

Benefits of classless approach:
- HTML documents are readable without CSS knowledge
- Accessibility correct by default through proper element usage
- CSS changes automatically apply everywhere an element appears
- Specificity stays low (element selectors)
- No class naming debates or conventions to enforce
- Smaller CSS without utility repetition

The primary objection is handling variations. How do you distinguish a "primary" button from a "secondary" button without classes? Several options:
- Context: buttons in nav versus buttons in forms
- Attributes: button[type="submit"] versus button[type="button"]
- Custom data attributes: button[data-variant="primary"]
- Custom elements: `<primary-button>` versus `<secondary-button>`

These options suffice for most needs. When they don't, a single class is acceptable. The goal is minimizing classes, not eliminating them entirely.

**Implications**

- Design phase must consider available semantic elements
- CSS organization follows element categories rather than component names
- Team must learn semantic element vocabulary
- HTML review should check for appropriate element usage
- Accessibility testing simplified but still required

**Related Decisions**

- ADR-002: No JavaScript Frameworks (no CSS-in-JS patterns)
- ADR-005: No Build Step (plain CSS, no preprocessing)

**Related Requirements**

- Application must be accessible to screen reader users
- Application must be indexable by search engines
- Styling must be maintainable without specialized tooling

**Related Artifacts**

**Related Principles**

- Accessibility over convenience
- Meaning over presentation
- Browser defaults as foundation

**Notes**

Team should review HTML5 element catalog before development. Many developers underutilize available elements (details/summary for accordions, dialog for modals, figure/figcaption for images with captions, time for dates, mark for highlights, etc.).
