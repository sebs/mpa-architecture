# ADR-005: No Build Step for Production

**Issue**

Modern web development typically involves build tooling: bundlers aggregate modules, transpilers convert modern syntax, minifiers reduce file sizes, and various optimizations prepare code for production. This tooling adds complexity, configuration burden, and creates divergence between authored and executed code. We must decide whether production deployment requires build processing.

**Decision**

Production deployment requires no build step. HTML, CSS, and JavaScript files serve as authored. ES modules load directly in browsers. Only standard HTTP compression (gzip/brotli) applies at the server level.

**Status**

Decided

**Group**

Infrastructure

**Assumptions**

- Target browsers support ES modules natively
- Target browsers support modern JavaScript syntax (ES2020+)
- HTTP compression provides sufficient size reduction without minification
- Module count is manageable without bundling (under 50 modules per page)
- Development team values debuggability over marginal performance gains

**Constraints**

- Cannot use TypeScript for runtime code (type checking as separate dev tool is acceptable)
- Cannot use JSX or non-standard syntax
- Cannot automatically tree-shake unused exports
- Cannot automatically polyfill for older browsers
- Module request count may impact initial load on slow connections

**Positions**

1. **Full build pipeline** — Bundler (webpack, Rollup, esbuild), transpiler (Babel), minifier, and various plugins. Maximum optimization but high configuration complexity and source map dependency. Rejected due to configuration complexity and source map dependency.

2. **Minimal bundling only** — Bundle modules to reduce request count but skip transpilation. Simpler than full pipeline but still adds build dependency. Rejected due to the build dependency.

3. **Development-only build** — Build during development for ergonomics but produce unbundled output for production. Confusing divergence between environments. Rejected due to divergence between development and production.

4. **No build step** — Serve source files directly. ES modules in browsers. Plain CSS. Native JavaScript. This is our selected position.

**Argument**

Build tools solve problems that may not exist in our context:

**Bundling** reduces HTTP requests. With HTTP/2 multiplexing and browser module caching, request count matters less than historically. A page loading 20 small modules may perform better than loading one large bundle if most modules are cached.

**Transpilation** enables modern syntax in older browsers. Our target browsers (last 2 years of evergreen browsers) support ES2020+ natively. Transpilation adds no value while obscuring code.

**Minification** reduces file sizes. Modern compression (brotli) on well-written code achieves similar results without transforming the code. The marginal size difference rarely affects user experience.

What we gain by eliminating build:
- View source shows actual code
- Browser DevTools line numbers match source files
- No source map configuration or debugging issues
- Errors reference real file locations
- No build configuration to maintain or debug
- No build tool version management
- Deployment is copying files
- New developers onboard without tooling setup

The debugging advantage is substantial. Framework and build tool issues consume significant development time. "It works in development but not production" disappears when they are the same.

**Implications**

- JavaScript must be valid ES2020+ syntax
- CSS must be valid modern CSS (custom properties, grid, etc.)
- Module organization must be intentional to manage request count
- Development server useful for local work (live reload, local HTTPS) but not required
- Linting and formatting run as development aids, not build steps

**Related Requirements**

- Application must be deployable via static file hosting
- Debugging must not require source maps
- Development setup must not require build tool installation

**Related Artifacts**

- example/package.json (no build script; test and static-serve only)
- readme.md (run instructions)

**Related Principles**

- Debuggability over optimization
- Simplicity over capability
- What you write is what runs

**Notes**

If performance testing reveals request count issues, consider manual module consolidation rather than introducing bundling. Grouping related functionality into fewer files is simpler than build tool configuration.

banira (component test mounting, documentation generation) is development tooling in the same category as linting and type checking. Production serves the authored JavaScript files unchanged.
