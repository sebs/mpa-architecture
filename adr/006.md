# ADR-006: IndexedDB for DuckDB Persistence

**Issue**

DuckDB-WASM operates in browser memory. When the page unloads, the database is lost. For the application to maintain data across sessions, browser restarts, and device reboots, the database must persist to durable storage. The browser provides several storage mechanisms with different characteristics.

**Decision**

Persist DuckDB to IndexedDB. The database serializes to a binary blob and stores as a single IndexedDB entry. On page load, check for existing data and restore if present.

**Status**

Decided

**Group**

Data

**Assumptions**

- Database size typically under 50MB serialized
- Full serialization performance is acceptable (under 500ms for typical sizes)
- Browser storage quotas sufficient for application needs
- Users accept browser-managed storage (can be cleared)
- Single database instance per origin is sufficient

**Constraints**

- IndexedDB storage can be cleared by user or browser under storage pressure
- Storage quotas vary by browser and available disk space
- Serializing entire database on each save has performance cost
- No incremental persistence; entire database written each time
- IndexedDB operations are asynchronous

**Positions**

1. **localStorage** — Synchronous, simple API. Limited to approximately 5MB. String-only storage requires base64 encoding of binary data. Rejected due to size limit.

2. **IndexedDB** — Asynchronous, supports binary data, generous quotas (hundreds of MB to GB depending on browser). More complex API but well-supported. This is our selected position.

3. **Cache API** — Designed for HTTP response caching. Could store database as fake response but semantically inappropriate. Rejected as misuse of API.

4. **Origin Private File System (OPFS)** — File system-like interface with potentially better performance. Newer API with less browser support. Promising but premature for production reliance.

5. **Remote server sync** — Store authoritative data on server, browser is cache. Adds server dependency. Rejected as contradicting offline-first architecture.

**Argument**

IndexedDB is the only browser storage mechanism meeting all requirements:
- Sufficient capacity (GB available with user permission)
- Binary data support (no base64 overhead)
- Durable across sessions
- Well-supported across browsers
- No server dependency

The IndexedDB API is notoriously awkward, but DuckDB-WASM provides serialization to ArrayBuffer, and storing a single binary blob in IndexedDB is straightforward. A small wrapper converts callbacks to promises for cleaner code.

Persistence strategy:
- On page load: check IndexedDB for existing database blob, restore to DuckDB if found
- On data modification: serialize DuckDB to blob, store to IndexedDB
- On page unload: ensure final persistence (beforeunload handler)

The "entire database each time" approach is simple but has scaling limits. For databases over 50MB, consider:
- Debouncing writes to avoid excessive serialization
- Background serialization via Web Worker
- Investigating OPFS when browser support improves

**Implications**

- Persistence code must handle missing database (first-run case)
- Must handle storage quota errors gracefully
- Users should understand data is local and can be lost
- Export/import functionality may be desired for backup
- Each page load incurs restore time; affects perceived performance

**Notes**

OPFS should be monitored for browser support improvements. It may offer better performance for large databases through incremental write capability. Safari support is the primary blocker as of this writing.
