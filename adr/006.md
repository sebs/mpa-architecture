# ADR-006: Data Durability and Export Strategy

**Issue**

IndexedDB is our primary data store, but browser storage is not permanent. Users can clear site data, browsers may evict storage under pressure, and device changes lose all local data. We must address data durability expectations and provide mechanisms for users to protect their data.

**Decision**

Accept IndexedDB as ephemeral storage with user-facing export/import capabilities. Users are responsible for backing up important data. The application provides clear export mechanisms and communicates storage limitations.

**Status**

Decided

**Group**

Data

**Assumptions**

- Users understand browser storage can be cleared
- Export/import functionality is acceptable for data portability
- Most user data is recoverable or non-critical
- Storage eviction is rare in practice for active applications
- JSON is an acceptable export format for structured data

**Constraints**

- IndexedDB storage can be cleared by user (clear site data)
- Browser may evict data under storage pressure (rare but possible)
- No automatic cloud backup without server infrastructure
- Cross-device sync requires manual export/import or server component
- Large exports may be slow to generate

**Positions**

1. **Accept ephemeral storage** — Document limitations, provide export/import. Simple, no server dependency. This is our selected position.

2. **Request persistent storage** — Use Storage API to request persistent storage. Reduces eviction risk but not elimination. User must grant permission.

3. **Server-side backup** — Sync to remote server. Reliable but adds server dependency and authentication complexity. Rejected as contradicting offline-first architecture.

4. **File system storage (OPFS)** — Use Origin Private File System for more durable storage. Limited browser support. API still maturing. Rejected as premature.

**Argument**

Browser storage is fundamentally different from server storage. Users should understand this distinction. Fighting it leads to false expectations and complex workarounds.

Our approach embraces reality:

**Transparency.** Users know their data lives in the browser. Clear messaging explains what this means.

**Export capability.** One-click export to JSON file. Users can save to cloud storage, email to themselves, or store locally. They control their backup strategy.

**Import capability.** Restore from previously exported JSON. Works across devices, browsers, and time.

**Graceful degradation.** If data is lost, the application still functions. Users can start fresh or restore from backup.

This approach has precedent in many browser-based tools. Users of note-taking apps, todo lists, and productivity tools understand local storage tradeoffs.

**Persistent Storage API consideration:**

The Storage API's `navigator.storage.persist()` requests that the browser not evict this origin's data. This is worth using but not relying upon:
- User must grant permission
- Browser can still honor "clear all site data"
- Not all browsers implement consistently
- Good defense-in-depth, not primary strategy

**Implications**

- Export/import UI must be prominent and easy to use
- Export format should be human-readable (JSON with clear structure)
- Application should prompt for export before known data loss scenarios
- First-run experience should explain storage model
- Consider periodic "reminder to export" for active users

**Related Decisions**

- ADR-003: IndexedDB as Primary Data Store (what we're protecting)
- ADR-007: Broadcast Channel (propagates imported data to other open tabs)
- ADR-008: Service Worker Caching (caches code, not user data)

**Related Requirements**

- Users must be able to export all their data
- Users must be able to import previously exported data
- Application must handle missing data gracefully
- Storage limitations must be clearly communicated

**Related Artifacts**

- Export/import implementation documentation
- User-facing data storage explanation
- Data format specification

**Related Principles**

- Honesty over false promises
- User control over automatic behavior
- Simplicity over complexity

**Notes**

Consider requesting persistent storage via Storage API as an enhancement. It provides an additional layer of protection at low implementation cost. Do not present it to users as a guarantee.

Future enhancement: automatic export to user's cloud storage (Google Drive, Dropbox) via their OAuth APIs. This adds complexity but solves the backup problem more elegantly for users who want it.
