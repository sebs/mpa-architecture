# ADR-007: Broadcast Channel for Cross-Tab Communication

**Issue**

Our MPA architecture means each browser tab runs independently with its own JavaScript context. Users may have multiple tabs open simultaneously on different pages. When data changes in one tab (e.g., adding item to cart on products page), other tabs (e.g., cart page) display stale information. We need a mechanism for tabs to communicate state changes.

**Decision**

Use Broadcast Channel API for cross-tab communication. Tabs subscribe to a named channel and broadcast messages when data changes occur. Messages describe what changed; receiving tabs query fresh data rather than applying deltas.

**Status**

Decided

**Group**

Integration

**Assumptions**

- Users commonly have multiple tabs open
- Data changes in one tab should reflect in others without manual refresh
- Same-origin restriction is acceptable (all tabs are same application)
- Message delivery does not need persistence (ephemeral notifications)
- Target browsers support Broadcast Channel API

**Constraints**

- Same-origin only; cannot coordinate across different domains
- No persistence; messages lost if no tab listening
- No guaranteed ordering for rapid message sequences
- Safari support only from version 15.4 (March 2022); older Safari lacks support

**Positions**

1. **localStorage events** — Storage events fire across tabs on localStorage changes. Limited to string data. Awkward API requiring data to live in localStorage. Rejected due to coupling with storage.

2. **SharedWorker** — Single worker shared across tabs, can coordinate messages. Complex lifecycle management. Not supported in some mobile browsers. Rejected due to complexity.

3. **Service Worker messaging** — Tabs communicate through Service Worker as intermediary. Requires round-trip through worker. More complex than needed for tab-to-tab messaging. Rejected due to unnecessary indirection.

4. **Broadcast Channel API** — Direct tab-to-tab messaging within origin. Simple pub/sub model. Structured object support. This is our selected position.

5. **Server-based coordination** — Tabs poll server or use WebSocket for updates. Adds server dependency. Rejected as contradicting offline-first architecture.

**Argument**

Broadcast Channel is purpose-built for same-origin tab communication. The API is minimal:
- Create channel: `new BroadcastChannel('name')`
- Send message: `channel.postMessage(data)`
- Receive message: `channel.onmessage = handler`
- Clean up: `channel.close()`

Messages are structured cloneable objects, not limited to strings. A cart update message can include product ID, quantity, and action type as a proper object.

Our synchronization pattern:
1. Tab modifies data (e.g., adds cart item)
2. Tab writes to IndexedDB
3. Tab broadcasts notification: `{ type: 'CART_UPDATED', itemId: 123 }`
4. Other tabs receive notification
5. Other tabs query IndexedDB for fresh cart data
6. Other tabs update their UI

This pattern avoids sending full state over broadcast (which could be large) and avoids complex delta reconciliation (which could have bugs). Each tab is responsible for querying its own view of the current state.

Safari 15.4+ requirement excludes some older iOS devices. For those users, tabs simply won't synchronize automatically; manual refresh still works. This degradation is acceptable.

**Implications**

- Message protocol must be defined (message types, payloads)
- Receiving tabs must have appropriate query handlers
- Tab lifecycle must handle channel cleanup
- Testing requires mock Broadcast Channel implementation
- Some older browsers require graceful degradation

**Notes**

Consider adding a visual indicator when tabs are synchronized versus potentially stale. A subtle "synced" icon provides user confidence. Also consider a manual refresh mechanism for users on unsupported browsers.
