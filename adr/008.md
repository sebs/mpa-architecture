# ADR-008: Service Worker Caching Strategy

**Issue**

Progressive Web App capability requires a Service Worker to intercept network requests and provide offline functionality. Different resource types benefit from different caching strategies—HTML needs freshness, WASM binaries need stability, assets need speed. We must define caching behavior before implementing the Service Worker.

**Decision**

Implement resource-type-specific caching strategies:
- HTML pages: network-first with cache fallback
- CSS and JavaScript: cache-first with version invalidation
- DuckDB WASM binary: cache permanently
- Images and static assets: cache-first

**Status**

Decided

**Group**

Infrastructure

**Assumptions**

- Offline functionality is a core requirement
- Different resources have different freshness requirements
- Cache storage has sufficient capacity
- Users accept that offline content may be stale
- Deployment process can coordinate cache version changes

**Constraints**

- Service Worker updates require deployment coordination
- Cache invalidation must happen reliably on updates
- Storage quota limits cache size
- Service Worker debugging adds complexity
- First visit cannot be offline (nothing cached yet)

**Positions**

1. **Cache-everything approach** — Cache all requests, serve cached always. Simple but risks stale content for frequently changing resources.

2. **Network-only approach** — Never cache, always fetch. Maximizes freshness but eliminates offline capability.

3. **Stale-while-revalidate for all** — Serve cached, fetch update in background. Good balance but may show stale HTML.

4. **Resource-specific strategies** — Different strategy per resource type based on freshness requirements. More complex but appropriate behavior per resource. This is our selected position.

**Argument**

Each resource type has different characteristics:

**HTML pages** may change between deployments. Users should see fresh content when online. Network-first ensures freshness while cache fallback enables offline. If network fails and no cache exists, show dedicated offline page.

**CSS and JavaScript** change only on deployment. Once cached, serve immediately for performance. Version changes through cache key naming (v1, v2, etc.) force fresh fetch after deployment.

**DuckDB WASM binary** is large (several MB) and changes rarely (only when DuckDB version updates). Cache aggressively, serve forever. Version in filename handles rare updates.

**Images and static assets** are immutable once deployed. Cache on first request, serve from cache thereafter. Content-addressable URLs (hash in filename) could ensure correctness but not required initially.

Cache versioning strategy:
- Cache name includes version: `app-cache-v1`
- Deployment bumps version: `app-cache-v2`
- Service Worker activate event deletes old caches
- HTML pages not included in versioned cache (network-first handles freshness)

**Implications**

- Deployment must update Service Worker cache version
- Service Worker registration must happen on each page
- Offline page must be designed and cached
- Cache size should be monitored
- Users may need to close all tabs for Service Worker update

**Related Decisions**

- ADR-001: Multi-Page Architecture (pages are cached resources)
- ADR-003: DuckDB as Primary Data Store (WASM binary caching critical)
- ADR-006: IndexedDB Persistence (complements Service Worker cache)

**Related Requirements**

- Application must function offline
- Cached content must update on deployment
- Large assets (WASM) must not re-download unnecessarily
- Offline state should be clearly communicated to user

**Related Artifacts**

- Service Worker implementation
- Offline page design
- Deployment checklist including cache version update
- Cache monitoring documentation

**Related Principles**

- Offline-first over network-dependent
- Performance through caching
- Graceful degradation for first visits

**Notes**

Consider implementing a "new version available" notification when Service Worker detects update. User can choose when to refresh for new version rather than having it forced.
