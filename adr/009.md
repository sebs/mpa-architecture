# ADR-009: URL Parameters as Primary Navigation State

**Issue**

Application state must be stored somewhere. SPAs typically use in-memory state management (Redux, Vuex, etc.). Our MPA architecture loses JavaScript context on navigation. We must determine where view configuration state—filters, sort orders, pagination, search queries—will reside to survive navigation, enable bookmarking, and support sharing.

**Decision**

URL parameters hold all navigation and view configuration state. State that should survive page reload, be bookmarkable, or be shareable encodes in the URL query string.

**Status**

Decided

**Group**

Presentation

**Assumptions**

- URL length limits (approximately 2000 characters) are sufficient for typical state
- State is not sensitive (can appear in browser history, server logs)
- Users benefit from shareable and bookmarkable filtered views
- Development team will design URLs thoughtfully
- State complexity per page is moderate (under 10 parameters typically)

**Constraints**

- URL length limited to approximately 2000 characters practically
- Complex nested state requires encoding scheme
- Sensitive information cannot go in URLs
- URL changes affect browser history; must manage carefully
- Server logs capture URL parameters (privacy consideration)

**Positions**

1. **sessionStorage** — Survives navigation within tab, lost on tab close. Not shareable or bookmarkable. Rejected due to sharing and bookmark limitations.

2. **localStorage** — Persists across sessions. Shareable state requires manual export. Not bookmarkable. Rejected due to bookmark and sharing limitations.

3. **DuckDB "preferences" table** — Store view state in database. Persists across sessions. Not shareable without export mechanism. Rejected due to sharing limitations.

4. **URL parameters** — Visible, shareable, bookmarkable, survives navigation. Standard web pattern. This is our selected position.

5. **URL hash fragment** — Alternative URL location. Not sent to server. Historically used for SPA routing. No advantage over query parameters for our use case.

**Argument**

URL parameters are the web's native mechanism for view state. Benefits:

**Shareability** — Copy URL, send to colleague, they see same filtered view. Zero friction sharing.

**Bookmarkability** — Save filtered view to bookmarks. Return later to exact state. Browser feature, not application feature.

**History integration** — Browser back button returns to previous filter state. Native behavior, no code required.

**Server visibility** — Analytics can track popular filter combinations. Useful for understanding user behavior.

**Debuggability** — View state visible in URL bar. No inspection tools needed to see current state.

URL design matters:
- Use descriptive parameter names: `?category=electronics` not `?c=1`
- Use human-readable values where possible
- Consider URL aesthetics; users see these
- Establish naming conventions early

State distribution:
- **URL parameters**: View configuration (filters, sort, page, search query)
- **IndexedDB**: Persistent user data (cart contents, preferences)
- **Neither**: Transient UI state (dropdown open, tooltip visible)

Avoid putting data in URL that belongs in database. Cart contents persist via IndexedDB, not URL. A URL might reference a cart view (`/cart.html`) but not encode cart contents.

**Implications**

- URL schema design is architectural work; invest upfront
- Parameter parsing needed on each page load
- History manipulation requires care (`replaceState` vs `pushState`)
- Some state naturally stays ephemeral (UI open/close state)
- Testing should verify URL parameter handling

**Related Decisions**

- ADR-003: IndexedDB as Primary Data Store (holds data, not view state)

**Related Requirements**

- Filtered views must be shareable via URL
- Bookmarks must restore exact view state
- Browser back button must return to previous view state
- URLs must be human-readable

**Related Artifacts**

- URL schema documentation

**Related Principles**

- URLs are user interface
- Shareable by default
- Browser features over application features

**Notes**

Consider URL parameter compression if length becomes an issue. Libraries exist to shorten parameter encoding while maintaining human-readability. Premature optimization to implement initially; note as future option.
