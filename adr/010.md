## ADR-010: Node.js Test Runner with jsdom

**Issue**

Testing browser-based code requires either actual browsers (slow, complex CI) or simulated browser environments in Node.js. We must select testing infrastructure that supports our vanilla JavaScript code, IndexedDB operations, and DOM manipulation while aligning with our no-build-step philosophy.

**Decision**

Use Node.js built-in test runner (Node 18+) combined with jsdom for DOM simulation. IndexedDB is simulated via fake-indexeddb for database tests. Browser-specific APIs use mock implementations where needed.

**Status**

Decided

**Group**

Infrastructure

**Assumptions**

- Development uses Node.js 18 or later
- jsdom provides sufficient DOM fidelity for our testing needs
- IndexedDB can be simulated in Node.js via fake-indexeddb
- Unit and integration tests do not require visual verification
- CI environment supports Node.js execution

**Constraints**

- jsdom is not a real browser; some APIs missing or imperfect
- Service Worker cannot run in jsdom; requires mocking
- Broadcast Channel requires mock implementation
- No visual regression testing without additional tooling
- ES module support requires appropriate Node.js configuration

**Positions**

1. **Jest** — Popular testing framework. Rich feature set. Adds dependency. Requires configuration for ES modules. Rejected to minimize dependencies.

2. **Mocha + Chai** — Established combination. Flexible. Multiple dependencies to manage. Rejected to minimize dependencies.

3. **Vitest** — Modern, fast, Vite-aligned. Assumes build tooling context. Rejected due to build tool assumption.

4. **Playwright/Puppeteer for all tests** — Real browser execution. Accurate but slow. Complex CI setup. Rejected as primary approach; acceptable for targeted integration tests.

5. **Node.js test runner + jsdom** — Built into Node.js. ES module support. jsdom provides DOM. Minimal dependencies. This is our selected position.

**Argument**

Node.js test runner (introduced Node 18, stable Node 20) provides:
- Built-in assertion library
- Test isolation
- Parallel execution
- Watch mode
- TAP and spec reporters
- No external dependencies

Adding jsdom provides DOM APIs: document, window, element creation, event dispatch. This combination covers most testing needs without external test framework dependencies.

Test categories and approaches:

**Database tests** — fake-indexeddb provides IndexedDB simulation in Node.js. Test object store operations, indexes, transactions, and queries. No DOM needed.

**State management tests** — Test URL parameter parsing, state merging. jsdom provides URL and Location APIs.

**DOM manipulation tests** — Test rendering functions. jsdom provides document APIs. Verify correct elements created with correct attributes.

**Broadcast Channel tests** — API not in jsdom. Implement simple mock that records sent messages and can trigger received messages. Test handler behavior.

**Service Worker tests** — Cannot run in Node.js. Test Service Worker in isolation with actual browser if needed. Mock fetch interception for unit tests.

For high-confidence integration testing, Playwright can run actual browser tests. Use sparingly for critical paths rather than comprehensive coverage.

**Implications**

- Node.js 18+ required for development
- jsdom and fake-indexeddb added as development dependencies
- Mock implementations needed for some browser APIs (Broadcast Channel, Service Worker)
- ES module configuration required in package.json
- Some tests may pass in jsdom but fail in real browsers (rare but possible)

**Related Decisions**

- ADR-002: No JavaScript Frameworks (no framework-specific test utilities)
- ADR-005: No Build Step (tests run against actual source files)
- ADR-007: Broadcast Channel (requires mock for testing)

**Related Requirements**

- Tests must run without build step
- Tests must execute in CI environment
- Tests must cover database operations, state management, and DOM updates
- Test execution must complete in reasonable time (under 2 minutes for full suite)

**Related Artifacts**

- Test organization documentation
- Mock implementation files
- CI pipeline configuration
- Test coverage goals

**Related Principles**

- Minimal dependencies
- Test actual code, not transpiled code
- Confidence over coverage percentage

**Notes**

If jsdom limitations become problematic, consider targeted Playwright tests for affected functionality. The goal is confidence in correctness, not rigid adherence to tooling choice.
